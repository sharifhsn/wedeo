# H.264 Baseline Decoder — Architecture & Status

## Overview

Native Rust H.264/AVC Baseline decoder (`codecs/wedeo-codec-h264/`) and Annex B
raw bitstream demuxer (`formats/wedeo-format-h264/`). ~15,400 lines of Rust,
309 unit tests, 0 clippy warnings. No unsafe code.

**Target**: Bit-exact decoded YUV420p output matching FFmpeg's FATE conformance
tests, verified via framecrc checksums.

**Current state**: I-frame and P-frame decode pipeline works end-to-end.
P-frame decode via `mb_skip_run` + P_SKIP + coded P-MB types (0-4) is wired.
4 of 17 Baseline conformance tests are BITEXACT (no-deblock). All 17 files
decode all frames. Frames 0-30 match for many P-frame files (BA_MW_D,
BANM_MW_D, AUD_MW_E, BA2_Sony_F). One root cause (intra4x4 neighbor modes
not propagated from inter MBs in P-slices) explains most remaining P-frame
diffs. Multi-slice I-frame corruption and BASQP1 CAVLC desync are separate
issues.

## Decode Pipeline

```
.264 file → Demuxer → Packets (one per access unit)
                          ↓
              Decoder::send_packet()
                          ↓
              split_annex_b() → NAL units
                          ↓
              ┌─ SPS → sps_list[0..31]
              ├─ PPS → pps_list[0..255]
              ├─ SEI → ignored (parsed for recovery_point)
              ├─ AUD → ignored
              └─ Slice (IDR/non-IDR):
                    ↓
                  parse_slice_header() → SliceHeader
                    ↓
                  decode_slice_into(FrameDecodeContext):
                    ↓
                  while mb_addr < total_mbs:
                    ↓
                  [P/B-slice] mb_skip_run = ue(v)
                    ├─ skip_run > 0: decode_skip_mb() × N
                    │    predict_mv_skip_full() → apply_mc_partition()
                    │    (no residual, no bitstream reads)
                    └─ coded MB:
                         decode_mb_cavlc()  → MacroblockCavlc
                           ↓                   (mb_type, pred modes, CBP, coefficients)
                         decode_macroblock():
                           ├─ I_4x4:
                           │    for each 4x4 block:
                           │      predict_4x4() → dequant_4x4() → idct4x4_add()
                           ├─ I_16x16:
                           │    predict_16x16()
                           │    luma_dc_dequant_idct() → per-block idct4x4_add()
                           ├─ P_L0_16x16 / 16x8 / 8x16 / 8x8:
                           │    predict_mv() → apply_mc_partition() → add residual
                           └─ chroma:
                                predict_chroma_8x8() (intra) or MC (inter)
                                chroma_dc_dequant_idct() → per-block idct4x4_add()
                    ↓
                  update neighbor context (nz counts, left/top)
                  store MbDeblockInfo
                    ↓
                  deblock_frame()  (after all MBs)
                    ↓
                  PictureBuffer → Frame (Y/U/V planes)
                    ↓
              Decoder::receive_frame()
```

## Key Data Structures

### MacroblockCavlc (cavlc.rs)
Holds all syntax elements parsed from one macroblock:
- `mb_type`, `is_intra`, `is_intra4x4`, `is_intra16x16`, `is_pcm`
- `intra4x4_pred_mode[16]` — per-block prediction direction
- `cbp` — coded block pattern (luma 8x8 bits + chroma DC/AC bits)
- `luma_coeffs[16][16]` — dequantized 4x4 block coefficients
- `luma_dc[16]` — I_16x16 DC coefficients (pre-Hadamard)
- `chroma_dc[2][4]` — chroma DC per plane
- `chroma_ac[2][4][16]` — chroma AC per plane per block
- `non_zero_count[24]` — context for next MB's nC computation

### FrameDecodeContext (mb.rs)
Per-frame state during decode:
- `pic: PictureBuffer` — Y/U/V planes being filled
- `mb_info: Vec<MbDeblockInfo>` — deblocking metadata per MB
- `neighbor_ctx: NeighborContext` — CAVLC nC context (top/left nz counts)
- `qp: u8` — current QP (modified per-MB by qp_delta)
- `dequant4: Dequant4Table` — pre-computed dequant coefficients
- `mv_ctx: MvContext` — per-4x4-block MV and ref_idx for neighbor lookup

### H264Decoder (decoder.rs)
Persistent decoder state:
- `sps_list[32]`, `pps_list[256]` — parameter sets
- `current_fdc` — in-progress FrameDecodeContext for multi-slice frames
- `output_queue` — decoded frames awaiting receive_frame()
- `nalff_length_size` — MP4/avcC NAL length size (0 = Annex B)
- `dpb: Dpb` — Decoded Picture Buffer for reference picture management
- `ref_list_l0: Vec<usize>` — reference list 0 (DPB indices) for current slice

## CAVLC Decode Algorithm (cavlc.rs)

The core coefficient decoding follows H.264 spec section 9.2:

1. **coeff_token** — VLC from one of 4 tables (selected by nC context):
   - nC 0-1, 2-3, 4-7: variable-length codes
   - nC >= 8: fixed 6-bit code
   - nC = -1: chroma DC table
   - Returns (total_coeff, trailing_ones)

2. **Trailing ones** — 1 sign bit each (1=negative), most recent first

3. **Remaining levels** — prefix (leading zeros) + suffix:
   - `suffix_length` starts at 0 (or 1 if total_coeff > 10 && trailing_ones < 3)
   - First non-trailing level gets +2 offset when trailing_ones < 3
   - `suffix_length` increases when |level| exceeds threshold
   - Level sign: even level_code → positive, odd → negative

4. **total_zeros** — VLC (15 tables for 4x4, 3 for chroma DC)

5. **run_before** — VLC per coefficient (7 tables by zeros_left)

6. **Coefficient placement** — placed from highest scan position downward

## Known Issues

### Demuxer access unit grouping
The H.264 Annex B demuxer currently groups all NALs from the file into a single
packet instead of splitting per-frame. The decoder handles this correctly by
tracking frame boundaries via `first_mb_in_slice == 0`, but timestamps are
wrong (all frames get the same packet PTS). The decoder works around this by
using its own frame counter for PTS.

### Pixel-level bitexact parity for non-BITEXACT files
Most Baseline conformance files decode all frames but have pixel-level
differences vs FFmpeg. 3 of 17 files are BITEXACT. Known issues:

**I-frame issues (2026-03-18 FIXED)**:
- Per-MB QP variation (BAMQ1): **FIXED** — IDCT pass order was wrong.
  FFmpeg uses a transposed (column-major) coefficient layout and its IDCT
  code effectively does row-transform-first, column-transform-second.
  Wedeo had column-first-then-row with row-major coefficients, causing ±1
  rounding diffs from the >>1 truncation in the butterfly. Fix: swap the
  two IDCT passes to match FFmpeg's effective computation order.
- QP=0 multi-slice (BASQP1): CAVLC parse failures in continuation slices.
  First slice (first_mb=0) decodes OK, subsequent slices fail with
  `InvalidData`. `run_before` values exceed `zeros_left` — bitstream
  position error within the slice. Root cause: likely a CAVLC level or
  total_zeros VLC decode bug triggered by specific coefficient patterns
  at low QP.

**P-frame issues**:
- Intra4x4 neighbor mode propagation from inter MBs: when a P-slice MB is
  I_NxN, its blk(0,0) needs the left/top neighbor's intra4x4 modes. If
  those neighbors are inter (P_SKIP, P_L0_*), their stored modes are `-1`.
  The H.264 spec says inter neighbor modes should be treated as DC_PRED=2,
  not "unavailable". This causes wrong predicted mode → wrong rem_mode
  decode → wrong pixels. First appears at frame 31 MB(5,5) in BA_MW_D,
  BANM_MW_D, AUD_MW_E (same bitstream position, same bug).
  **Root cause identified, fix pending.**

## Known Remaining Issues

1. **Intra4x4 neighbor modes from inter MBs** — inter MB neighbors store
   mode=-1 in `left_intra4x4_mode`/`top_intra4x4_mode`, causing the predicted
   mode to default to DC_PRED when it should also be DC_PRED but via a
   different code path (left_mode=2 vs left_mode=-1 both yield predicted=2,
   BUT when left_mode is -1 it triggers `predicted = DC_PRED` unconditionally
   instead of `min(left, top)`). **Identified via trace at bit_pos=1483.**
   Affects BA_MW_D, BANM_MW_D, AUD_MW_E, BA2_Sony_F — fixing this may make
   these files BITEXACT.

2. **Multi-slice I-frame corruption** — SVA_Base_B, SVA_FM1_E, SVA_CL1_E have
   66/99 MBs wrong from frame 0. First correct MBs are rows 0-2 (MB 0-32),
   corruption starts at MB(0,3). Likely a slice boundary context reset issue
   where the continuation slice doesn't properly inherit neighbor state.

3. **BASQP1 multi-slice CAVLC desync** — QP=0, 5 MBs per slice. First slice
   decodes correctly, continuation slices (first_mb=5,10,...) produce garbage
   (94/99 MBs wrong, max_diff=245).

4. **BA1_FT_C 352x288** — nearly all MBs wrong from frame 0. May be a
   stride/dimension issue at the larger resolution, or could share root cause
   with multi-slice issues.

5. **Early P-frame diffs** (BAMQ2, SVA_BA2_D, SVA_NL2_E, BA3_SVA_C) — diffs
   appear at frame 1-2, different pattern from the frame-31 intra4x4 bug.
   May be a different MV prediction or MC issue.
